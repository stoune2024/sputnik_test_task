## Решение тестового задание на позицию Backend разработчика (Python)

**1. Архитектурная проблема (самая главная): Вся логика работы с сущностями File и Alert перемешана в одном слое backend/src   ❌**


Это нарушает первый принцип SOLID -  принцип единственной ответственности (SRP).

**Решение:**

Выделение двух сущностей File и Alert в отдельные два слоя - /file_service и /alert_service сооответсвенно

**2. Проблема: Подключение и работа с БД внутри service.py  ❌**


service.py - бизнес логика

repository.py - работа с БД

Итого: нарушаем SRP


**Решение:**

Создание отдельного модуля repository в корне backend

Таким образом, соблюдаем SRP


**3. Проблема: app.py работает и с File, и с Alert, и с Celery  ❌**

Таким образом, нарушаем SRP. Более того, на практике такие модули называют controllers.py, а маршрутизацию запросов в отдельный слой.

**Решение:**

В каждом из слоев создаем controller.py с соответствующими ручками.

Запросы к app = FastAPI() маршрутизируем в file_router и alert_router в /backend/routers

backend/routers/api_v1_router.py становится единым центром подключения новых слоев/сервисов

С ним удобно версионировать API и настраивать подпути из одного места

**4. Проблема: schemas.py, models.py - нарушение SRP❌**

**Решение:**

models.py переносим в корень бекенда, а pydantic-cхемы в каждый из слоев

**5. Проблема: бизнес логика File и Alert в одном service.py - нарушение SRP❌**

**Решение:**

Распиливаем бизнес логику в каждый из слоев

**6. Проблема: tasks.py - нарушение DRY❌**

Он создает engine и sessionmaker, хотя они уже созданы

**Решение:**

Импортируем из repository.py


**7. Проблема: tasks.py - магические строки❌**

Встречаются десятки строк:

"processing"

"uploaded"

"processed"

"failed"

"suspicious"

"clean"

**Решение:**

Заменяем на Enum
